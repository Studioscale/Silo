## 1. System Overview

Silo has three tiers of storage:

| Tier | What | Principle | Contents |
|------|------|-----------|----------|
| **Topic files** | Domain-specific knowledge | "What is true NOW" | Curated facts with source material |
| **Event log** | Tagged daily entries | "What happened and WHEN" | One-liners with context and rationale |
| **Session transcripts** | Raw conversation exports | "What was said" | Platform-specific format (read-only) |

The AI assistant auto-loads a small set of files every session (~5-8 KB):
- **Operating rules** (the equivalent of AGENTS.md — how to behave)
- **Topic index** (one-line summary per topic file — the router)
- **Today's event log** (what happened today)

Everything else is loaded on demand — either by the user asking for it, or by
the assistant suggesting a relevant topic file to load.

### What the assistant does NOT do

- Does not read Layer 3 of topic files directly (uses search tools)
- Does not auto-load topic files (suggests, waits for confirmation)
- Does not run curation (scripts handle curation)
- Does not modify the topic index (auto-generated by script)
- Does not set validator-managed header fields

## 1a. v12.5 — Operation Log + Projections

The original Silo design (described throughout the rest of this document) treated topic files as the source of truth: scripts wrote to topic files directly; curation reorganized them in place; the file system *was* the memory.

The v12.5 implementation in [`src/`](src/) inverts that: a **canonical operation log** is the source of truth, and topic files / event logs are **regenerated projections** of the log.

### Three zones

| Zone | What lives here | Owned by |
|------|-----------------|----------|
| **A — Authority** | The operation log (`/.silo/`). Append-only sequence of events, JCS-canonicalized + SHA-256 hash-chained. | Silo (CLI / MCP bridge writers) |
| **B — Projections** | Topic files, event log files, topic index. Regenerated from Zone A by `silo regenerate`. | Silo (regenerator) |
| **C — Index / Cache** | BM25 search index, vector embedding index. Built from Zone B. | Memory search engine (e.g., OpenClaw's) |

Reads can hit any zone. Writes only ever land in Zone A. Anything Zone B or Zone C says is derived — if you don't trust it, you regenerate from the log.

### Write path (post-v12.5)

```
client (Claude / agent / cron / extraction pipeline)
   │
   │  write_event(slug, tag, content, principal, ...)
   ▼
admission gate (Event Capability Matrix — § 4 below)
   │
   │  if (socket, mode) cell admits this event type:
   ▼
log append: { seq, type, payload, hash_prev, hash_self, ts }
   │
   │  fsync, in-process mutex
   ▼
state: interpret(log) → topic_index, ACL table, etc.
   │
   ▼
regenerate Zone B files (atomic write per file)
```

### What this changes vs. the v3.x design

- **Replayability.** Any historical state is reachable via `interpret(log, asOfSeq=N)`. The file system at any past moment can be reconstructed exactly.
- **Byte-perfect regeneration.** A topic file regenerated from the log matches the original byte-for-byte (the v12.5 production migration measured 22/23 topics + 20/21 event logs as exact matches; the two diffs were one YAML field position and one trailing newline).
- **Single audit trail.** Every change is an event with a sequence number, hash, and intent_id. There's no parallel "who edited the file" question — the log is the answer.
- **No partial state.** Atomic file regeneration means readers never see half-written topic files. (Even at the point of a crash, the log is the truth; on restart, regenerate.)

### What this does NOT change

The conceptual layers (1/2/3 in topic files), the event log format, the search hierarchy, the extraction and curation pipelines, the design principles — all of those carry over from earlier specs. The change is **where the truth lives**, not **what the truth looks like**.

So sections 2 through 12 below describe the *file-shape* that's identical in both designs. The implementation just produces that file-shape from a log instead of from in-place edits.

### Event types (selected)

The v12.5 Event Capability Matrix is in [`src/matrix/matrix.yaml`](src/matrix/matrix.yaml). Examples of state-bearing events:

| Type | Effect |
|------|--------|
| `write_event` | Append to topic; payload carries `slug`, `tag`, `content`, optional `confidence` |
| `TOPIC_METADATA_SET` | Set/update topic header (type, tags, entities, status, summary) |
| `TOPIC_VERIFIED` | Mark topic as verified at this timestamp |
| `TOPIC_CURATED` | Mark topic as curated at this timestamp |
| `ACL_SEALED` | Set the reader set for a topic (admin-only) |
| `PRINCIPAL_DECLARED` / `_UID_BOUND` / `_ACCESS_ENABLED` | Identity events |
| `FEATURE_ACTIVE` / `_ROLLED_BACK` | Feature flag plane |
| `RECOVERY_MODE_ENTERED` / `_EXITED` | Mode plane |

Each event type has a (socket, mode) admission cell in the matrix. The interpret layer is a deterministic, total function over the log — same bytes always produce the same state.

## 2. Topic Files — The Three Layers

Every topic file has three physically separated sections.

### Layer 1 — Header (Manifest)

YAML frontmatter at the top of the file:

```yaml
---
topic: project-name
type: project                    # project | hobby | business | personal | reference
tags: [crm, flask, api, sales]
entities: [Ana, Ricardo, Acme Corp]
status: active                   # active | paused | archived | reference
created: 2026-01-15
last_verified: 2026-03-28       # When facts were last confirmed accurate
last_curated: 2026-03-28        # When curation last processed this file (script-owned)
curated_lines: 47               # Line count of Layer 2 (validator-owned)
source_lines: 312               # Line count of Layer 3 (validator-owned)
source_kb: 18.4                 # Size of Layer 3 in KB (validator-owned)
summary: >
  Custom CRM system replacing Pipedrive. Flask/SQLite backend with
  WhatsApp integration for quoting and sales pipeline.
---
```

**Field ownership matters.** `curated_lines`, `source_lines`, and `source_kb` are
computed by the file validator — never set manually. `last_curated` is set by the
curation script. The assistant and users own everything else.

### Layer 2 — Curated Facts

Between `<!-- CURATED_START -->` and `<!-- CURATED_END -->` markers.

- Target: 50-150 lines of distilled, structured knowledge
- Organized with markdown headers (##, ###)
- Updated **in place** when facts change
- Every change MUST have a changelog entry
- This is what the assistant loads when the topic is relevant

```markdown
<!-- CURATED_START -->

## Current State

- Backend: Flask/SQLite, deployed on Hetzner VPS
- Status: MVP complete, onboarding first 3 users
- WhatsApp integration: webhook-based, 500+ msgs/day capacity

## Key Decisions

- Chose Flask over Django (too heavy for our scale)
- SQLite over Postgres (single-server, no replication needed yet)

## Changelog

- 2026-03-28: [status] MVP complete. Was: in development. First users onboarding.
- 2026-03-15: [whatsapp] Switched to webhook. Was: polling. Polling too slow at scale.

<!-- CURATED_END -->
```

### Layer 3 — Source Material

Between `<!-- SOURCE_START -->` and `<!-- SOURCE_END -->` markers.

- Raw conversation excerpts and detailed context
- Each block has a mini-header: `### YYYY-MM-DD — Title` + blockquote summary
- Newest entries at the top (reverse chronological)
- Searchable via BM25 and semantic search
- **Never read directly by the assistant** — search tools find content here

```markdown
<!-- SOURCE_START -->

### 2026-03-28 — CRM MVP launch discussion
> Discussed go-live timeline, first user onboarding, and WhatsApp webhook performance.

User: How's the CRM looking for launch?
Assistant: The MVP is complete. Flask backend is stable, WhatsApp webhook
handles 500+ messages/day in testing...
(rest of conversation)

<!-- SOURCE_END -->
```

### Archive

When Layer 3 exceeds ~30 KB, oldest blocks move to `[slug].archive.md`. Archives
are searchable but never loaded or curated — frozen records.

## 3. Topic Index — The Router

Auto-generated by a script that reads all topic file headers. One compressed line
per topic:

```
project-name | project | crm,flask,api,sales | active | Custom CRM system replacing Pipedrive.
workshop | hobby | workshop,tools,cnc,welding | active | Workshop equipment and projects.
team | business | contacts,team,hr | active | Team members and roles.
```

**How the assistant uses it:** Already in context at startup. When conversation
touches a keyword, the assistant scans the index, identifies the relevant topic
file, and suggests loading it: "That touches the CRM file. Want me to pull it up?"

**How it stays current:** Regenerated as the final step of every script that
modifies topic files.

## 4. Event Log

### Format

One structured line per entry in `events/YYYY-MM-DD.md`:

```
[TAG] topic-slug: Content with enough context to stand alone.
```

### Tags

| Tag | When to use | Example |
|-----|-------------|---------|
| `[FACT]` | New information learned | `[FACT] team: Ana promoted to lead developer. Was: senior. Effective 2026-05.` |
| `[DECISION]` | A choice was made | `[DECISION] acme-crm: Chose webhook-based WhatsApp sync. Polling too slow.` |
| `[CHANGED]` | A known fact changed | `[CHANGED] workshop: CNC mileage 3,800 -> 4,200 hours. Routine check.` |
| `[TODO]` | Action item identified | `[TODO] business: Follow up with accountant re Q2 filing. Due 2026-04-15.` |
| `[EVENT]` | Something happened | `[EVENT] acme-crm: First customer onboarded. WhatsApp webhook live.` |
| `[PROCEDURE]` | A process was established | `[PROCEDURE] deploy: Staging test required before production push.` |

### Auto-extracted entries

The extraction pipeline writes entries with an `[AUTO-TAG:CONFIDENCE]` prefix:

```
[AUTO-FACT:CONFIRMED] team: Ana promoted to lead developer.
[AUTO-DECISION:TENTATIVE] acme-crm: Considering per-user pricing. No decision yet.
[AUTO-FACT:CONTEXT] market: Competitor launched a similar product last week.
```

Confidence levels:
- **CONFIRMED** — Explicit decisions or verified facts
- **TENTATIVE** — Considerations, inclinations, open questions
- **CONTEXT** — Background information, not a discrete decision

### Context standard

"Enough context" means someone reading only this one line, months later, with no
conversation history, understands what happened, why, and what was rejected.

## 5. Search Hierarchy

Always follow this order — cheapest first:

| Priority | Method | Cost | Best for |
|----------|--------|------|----------|
| 1 | Current context | Free | Always check first |
| 2 | Topic index scan | Free | Identify which topic file to load |
| 3 | Loaded topic file Layer 2 | Free | Already in context |
| 4 | Today's event log | Free | Already loaded |
| 5 | BM25 keyword search | Free | Exact terms, names, IDs, versions |
| 6 | Semantic search | ~1-3c | Conceptual queries, fuzzy matching |
| 7 | Ask one clarifying question | Free | Last resort |

Never jump to step 6 without trying step 5 first.

## 6. Extraction Pipeline

Runs on a schedule (e.g., every 30 minutes) and at session end.

### What it does

1. Reads session transcript deltas (only new lines since last run)
2. Calls an LLM (GPT-4o recommended) to extract facts, decisions, and changes
3. Assigns confidence levels (CONFIRMED, TENTATIVE, CONTEXT)
4. Assigns topic slugs from the topic index
5. Deduplicates against existing event log entries (BM25 Jaccard, threshold 0.8)
6. Appends surviving entries to today's event log

### Key design choices

- **Delta-based:** Tracks `lastProcessedLine` per session file. Never re-reads old content.
- **Coarse dedup on purpose:** Jaccard 0.8 catches near-exact duplicates. Semantic duplicates pass through — nightly curation handles the rest.
- **Unmatched facts use `general` slug:** When no topic fits, the entry goes to `general`. The topic suggestion pipeline detects clusters of general-slug entries.
- **Non-fatal:** Errors are logged, never propagated. The next scheduled run catches anything missed.

### Cost

~$0.02-0.08 per session (GPT-4o, depending on session length). ~$2-4/month at typical usage.

## 7. Curation Pipeline

Runs nightly. Only processes files that changed since the last curation run.

### Pre-flight (zero cost)

1. Scan all topic file headers
2. Compare file modification time vs `last_curated` for each
3. Build list of files that actually need curation
4. If list is empty — done, zero cost tonight

### Curation (tokens spent only on modified files)

For each modified file:
1. Review new entries in Layer 3
2. Extract facts/decisions into Layer 2 (update in place with changelog)
3. Update header metadata (entities, summary)
4. Propose new tags (logged for review, not auto-applied)
5. Check event log entries for matching `[CHANGED]` and `[FACT]` tags — propagate if Layer 2 is stale
6. Flag contradictions, vague changelogs, and structural issues

### Post-flight (zero cost)

1. File validator runs on all modified files
2. Topic index regenerated
3. Run summary appended to health log

### What the curation agent does NOT do

- Does not restructure or reorganize Layer 2 beyond specific fact changes
- Does not remove existing facts unless clearly superseded
- Does not auto-apply proposed tags (logged for human review)
- Does not touch Layer 3 content
- Does not modify validator-owned header fields

### Cost

~$0.01-0.03 per file (GPT-4o-mini). Typical nightly run: ~$0.02-0.06. Zero-file nights: $0.00.

## 8. Topic Suggestion Pipeline

Runs nightly, after curation.

### Problem it solves

When facts accumulate under the `general` slug, they indicate subjects that
deserve their own topic files. This pipeline detects clusters and suggests
creating new files.

### How it works

1. Scans event log entries from the last 30 days for `general`-slug entries
2. If fewer than 3 entries, exits (zero cost)
3. Calls an LLM to group entries by subject (filtering against existing topics)
4. Validates groups: minimum 3 entries, at least 1 FACT or DECISION
5. Writes suggestions to a bounded JSON signal file
6. The assistant surfaces suggestions to the user at the next session
7. On approval, a migration script creates the topic file

### The migration script

- Reads metadata and entries from the signal file (no LLM call)
- Creates the topic file with Layer 2 content grouped by entry type and confidence
- Populates Layer 3 with original event log lines as source material
- Validates the file, regenerates the topic index
- Marks the suggestion as created in the signal file

### Dismissed suggestions

When the user dismisses a suggestion, a `sourceMaxDate` (the newest evidence date)
is stored. Future detection runs ignore old evidence below this date — only genuinely
new entries can re-trigger the suggestion.

### Cost

$0.00 most nights. ~$0.01-0.02 when general-slug entries exist (one GPT-4o-mini call).

## 9. Auto-Load Budget

The assistant's startup context should be small and relevant:

| File | Size | Purpose |
|------|------|---------|
| Operating rules | ~2-3 KB | How to behave (search hierarchy, event log format, etc.) |
| Topic index | ~2-3 KB | One-line per topic file — the router |
| Today's event log | ~1-2 KB | What happened today |
| **Total** | **~5-8 KB** | Leaves 95%+ of context for conversation |

Everything else is loaded on demand. A loaded topic file (Layer 2 only) adds ~2-5 KB.

## 10. File Ownership and Permissions

If running in a containerized environment (Docker/OpenClaw):

- The container runs as a non-root user (e.g., uid 1000)
- Files written from outside the container must match this ownership
- The file validator preserves original modification time after header-only writes
  (prevents metadata updates from triggering false curation runs)

## 11. Health Monitoring

Every script run appends an entry to a health log:

```markdown
## 2026-04-06 02:15 — curation-runner
- Files needing curation: 2 (acme-crm, team)
- Actions: extracted 3 facts, updated 2 headers
- Fallbacks fired: 0
- Errors: [none]
- Duration: 12s
- Estimated cost: ~$0.04
```

### What to watch for

| Signal | Meaning | Action |
|--------|---------|--------|
| Curation not running | Cron broken or lock file stuck | Check cron config |
| Large error counts | Validator catching structural issues | Check recent manual edits |
| Zero-extraction warnings | Sessions with content but no entries | Review extraction prompt |
| Stale `last_verified` dates | Facts may be outdated | Schedule a verification pass |

## 12. Design Principles

1. **Curated truth over raw accumulation.** Layer 2 is maintained knowledge, not a dump. Every change is tracked.

2. **Cheapest search first.** Don't call a paid API when the answer is already in context.

3. **Scripts handle curation, the assistant handles conversation.** Clean separation of concerns. The assistant never runs curation tasks.

4. **Zero cost on quiet nights.** Pre-flight checks prevent unnecessary LLM calls. Most pipeline runs cost nothing.

5. **Fail safe, not fail silent.** Every script logs to the health log. Errors are visible. Nothing crashes the system.

6. **The event log is the staging layer.** Unmatched facts land in the event log, not in a separate inbox. The event log is simple, append-only, searchable, and already integrated with everything.

7. **User approval required for structural changes.** Topic files are never auto-created. The assistant suggests, the user decides.

8. **Human-readable everything.** No databases, no binary formats. Every piece of memory is a markdown file you can open in any text editor.
